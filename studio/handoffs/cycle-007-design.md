# Cycle 7 — Design

## Item
BACKLOG-005 [ai] WebLLM-backed brain — Qwen2.5-0.5B, lazy-loaded on first dialog.

## Why this cycle
This is the feature the whole project exists for: dinos whose words are generated by a local LLM in the browser, not picked from a canned list. Everything is in place — the `NPCBrain` boundary (004), personality to prompt with (010), a cast to speak (017). This cycle makes `makeBrain('webllm')` real and wires the dinos to it, with the stub demoted to a safety net.

## What ships
A `WebLLMBrain` implementing `NPCBrain`. The first time the player greets any dino, the brain begins loading Qwen2.5-0.5B in the background (via a dynamic import of `@mlc-ai/web-llm`, so it's code-split and boot stays instant). While the model loads — or if WebGPU/model are unavailable — `respond()` returns the existing canned stub text immediately, so dialog never blocks or breaks. Once the model is ready, greets return text the model generated from a prompt built out of the dino's name, species, and personality phrase. The dinos in the world use this brain (stub remains as the fallback path inside it).

Dev hooks (mirror `__clockNow`): `window.__brainStatus()` → `'idle' | 'loading' | 'ready' | 'fallback'`; greeting still flows through the existing Z dialog.

## Acceptance criteria
- [ ] `makeBrain('webllm')` returns an object implementing `NPCBrain` (has an async `respond`) and does NOT throw (unit — replaces the old "throws until BACKLOG-005" test).
- [ ] `buildMessages(ctx, obs)` (pure) produces a system message containing the dino's name, species, and personality phrase, and a user message reflecting the observation kind; total prompt is bounded (system+user, no unbounded growth) (unit).
- [ ] Before the model is ready, `WebLLMBrain.respond()` resolves to a non-empty reply (canned fallback) rather than hanging or throwing (unit — no WebGPU in Node).
- [ ] A forced load failure puts the brain in `fallback` status and `respond()` still returns a non-empty reply (unit).
- [ ] Reply text is trimmed to ≤ 200 characters regardless of source (unit).
- [ ] WebLLM is imported only under `game/src/ai/` — no scene or entity imports `@mlc-ai/web-llm` (unit/grep-style assertion or code review; boundary intact).
- [ ] Game boots with WebLLM dinos and `__brainStatus()` is a defined status string; greeting a dino returns a reply (e2e — in headless this stays on the fallback path, which is the point).
- [ ] No regression: clock, day/night, save, roster, hearts all still pass (e2e suites green).
- [ ] `npm run build` clean; vitest + playwright green.
- [ ] **Live inference** (model actually generating text) verified manually in a real browser with WebGPU — recorded in the QA handoff. (Not automatable headlessly.)

## Out of scope
- Device probe / model-size selection by VRAM (BACKLOG-006) — this cycle hardcodes the 0.5B model; 006 picks the size later.
- Memory, daily plans, reflection (011/012/014) — the prompt uses only static context this cycle.
- NPC-to-NPC dialogue (018).
- Streaming token-by-token display; a UI "thinking…" spinner (the instant canned reply covers the wait).
- Caching the model across sessions beyond whatever WebLLM/the browser does by default.

## Constraints
- **Hard boundary:** all WebLLM-specific code lives in `game/src/ai/`. No import of `@mlc-ai/web-llm` outside that folder. This is the charter's swap-to-native guarantee.
- The brain must **never throw out of `respond()`** and must **never block the dialog** — fallback to canned text on any error or while loading. Errors go to `console.error` (no silent failures).
- Pure prompt construction (`buildMessages`) must be in a no-WebLLM, Node-testable seam so the gate can verify it without a model.
- Reuse `describePersonality` (010) for the prompt and the existing `cannedGreetings`/`moodFromTraits` (brain.ts) for the fallback — do not duplicate either.
- Dynamic `import('@mlc-ai/web-llm')` inside the lazy init only (code-split, and import failure is catchable).
- `@mlc-ai/web-llm` is the only new dependency (already installed, charter-sanctioned). TypeScript strict; `any` only via the documented dev-hook pattern (and any unavoidable WebLLM typing, commented).
